# AGENTS.md

以下规范适用于当前仓库内的所有操作。

## Git 操作原则

- 谨慎执行 `git pull` / `git push`，每次需要向人类确认。
- 严格禁止直接向 `main` 提交 commit 或 `git push`，应当在分支执行开发。
- 不得修改已有的 commit，所有变动必须通过新的 commit 提交。
- 仅提交本次实际修改的文件；忽略工作区内与本次任务无关的变更，不需要处理或清理。
- 所有操作必须遵循本指南；若受限无法直接执行某个步骤，应整理可执行方案/命令由人类落地。

## Issue 管理与协作

### 触发与创建

- 开发类需求：
  - 先用 `gh issue list --state open` 等命令确认是否已有 open Issue 覆盖。
  - 若没有匹配项，必须立即创建新的 Issue 跟进，并把链接同步给人类。
- 文档/流程类需求：可直接执行，无需额外创建 Issue；若文档更新会引出后续开发，则按上条规则补建。
- 创建 Issue 时必须附上当前工作目录的 `git rev-parse HEAD` 结果，便于定位快照与追溯上下文。

### 执行与切换

- coding agent 任一时刻只能专注处理单一 Issue。
- 如需切换至其他 Issue：
  - 先检查当前 Issue 是否存在你引入且未提交的变更，若有必须先完成 git commit。
  - 若进展/评估/建议尚未追加到 Issue，则补评论并完成收尾，再切换到新的 Issue。
- 若发现重复或失效的 Issue，不得自行关闭；需先与人类确认，再按指示归档或关闭，以避免信息孤岛。
- 更新 Issue 前必须先阅读现有描述与所有评论，避免重复信息或反复陈述同一结论。
- 当前 Issue 取得实质进展或出现与既有结论不同的新评估时，应及时追加评论同步；同时仍需避免重复表达。

### 标题

- 使用一句话精准概括问题或需求，动词用现在时并说明触发条件/影响范围；必要时附版本号。
- Issue 标题前缀必须使用 `[feat]`、`[bug]`、`[docs]`、`[ops]`、`[chore]` 等主仓约定前缀（可按需要补充模块信息）。
- 避免模糊词或笼统描述。

### 标签

- 维护标准标签集，避免临时/重复标签；定期清理未使用标签并保证含义一致。
- 类型统一使用标题前缀（如 `[feat]`/`[bug]`/`[docs]`），不需要额外添加类型标签。
- 子仓约定：暂不处理使用 `inbox`；当前重点推进额外加 `focus`。

### 描述结构

- 描述应包含：背景 / 复现步骤 / 预期 vs 实际 / 验收标准 / 附件（按需取舍但要完整覆盖关键要点）。
- 链接、日志、截图需可复用且可访问，并保持命名一致。
- 涉及安全或业务风险时，补充影响评估与缓解建议。

### 验收标准

- 写成可验证条目，覆盖输入条件、行为与输出。
- 明确验收所需环境、账号、数据或访问权限；无法自动化时给出人工 checklist 与责任人。

### 关联与追踪

- 关联条目使用 `Closes #` / `Relates to #` 等语法；同仓引用使用 `#`，跨仓用完整链接。
- 在 Issue 或 PR 评论中引用同仓库条目必须使用 GitHub 自动链接语法（例如 `#56`、`#57`）；跨仓引用必须使用完整链接（示例：`https://github.com/Intelligent-Internet/commonground-next-a2a/issues/73`）。
- 若当前 commit 服务于某个 Issue，必须在 commit message 中标记对应 Issue（例如 `#20`）。
- PR 描述必须附上关联 Issue/任务链接，并说明依赖/阻塞关系（必要时用 checklist 管理子任务）。

## 工具与格式

- 读写 Issue 或 PR 必须使用 `gh` 命令行工具；不得直接通过网页进行操作。
- Issue/评论必须使用简体中文撰写；专业术语可保持英文原文，但不要混杂其他语言。
- 使用 `gh issue/comment create` 或 `gh issue/comment edit` 时，禁止在 `--body` 中直接写 `\n`；
  必须先将 Markdown 写入临时文件（如 `/tmp/body.md`），再通过 `--body-file` 传入，确保换行/列表正常渲染。
